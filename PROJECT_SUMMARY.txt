LINUX SYSTEM SRE ENVIRONMENT - PROJECT SUMMARY
================================================================================

PROJECT OVERVIEW
================================================================================
Linux System SRE Environment is an agentic training platform where AI agents
learn to solve real-world System Reliability Engineer (SRE) problems through
structured terminal interaction.

KEY INSIGHT:
This project demonstrates true "agency" - AI agents don't just respond to
queries; they actively troubleshoot infrastructure problems, set goals, plan
commands, observe results, and learn. This aligns with judge expectations
for "agentic" AI systems.

CORE COMPONENTS
================================================================================

1. VIRTUAL FILESYSTEM (virtual_filesystem.py - 196 lines)
   ─────────────────
   - Pure Python simulation of Linux filesystem
   - NO actual file system calls (100% safe)
   - Supports: /var/log/, /etc/, /proc/, /home/user/scripts/
   - Features:
     * Read/write file content
     * Manage file permissions (chmod)
     * Track processes (ps, kill, restart)
     * Instant reset to initial state
   - Advantages:
     * Perfect reproducibility
     * No side effects on host system
     * Instant creation/destruction
     * Easy to verify final state
   
   Key Classes:
   - VirtualFileSystem: Main filesystem manager

2. TERMINAL EMULATOR (terminal_emulator.py - 285 lines)
   ──────────────────
   - Shell command parser and executor
   - Restricted command set for safety and determinism
   - Supported commands:
     * cat (file reading)
     * grep (pattern matching)
     * ls (directory listing)
     * chmod (permission management)
     * ps (process listing)
     * kill (process termination)
     * systemctl (service management)
     * test (conditional checks)
     * echo, pwd (utilities)
   
   Design Principle:
   - Enough power for meaningful tasks
   - Safe (no destructive commands like rm)
   - Deterministic (no randomness)
   - Verifiable (clear input/output)
   
   Key Classes:
   - TerminalEmulator: Command parser and executor

3. TASKS (tasks.py - 268 lines)
   ─────
   Three difficulties with progressive grading:
   
   a) EASY: Log Analysis
      • Objective: Find timestamp of first 500 error in /var/log/app.log
      • Commands: cat, grep
      • Time to solve: ~1-2 minutes
      • Grading: Binary (0.0 or 1.0)
      • Key Skill: Basic text processing
   
   b) MEDIUM: Permission Repair
      • Objective: Make /home/user/scripts/cleanup.sh executable
      • Commands: chmod, ls
      • Time to solve: ~2-3 minutes
      • Grading: Staged (0.0, 0.5, 1.0)
      • Key Skill: File permissions understanding
      • Stages:
        - Stage 1: Script located (0.0)
        - Stage 2: Permission check attempted (0.5)
        - Stage 3: Script made executable (1.0)
   
   c) HARD: Process Recovery
      • Objective: Restart dead postgres service
      • Commands: ps, kill, systemctl, grep
      • Time to solve: ~3-5 minutes
      • Grading: Progressive (0.2, 0.5, 1.0)
      • Key Skill: Process management and troubleshooting
      • Stages:
        - Stage 1: Process status checked (0.2)
        - Stage 2: Process verified dead (0.5)
        - Stage 3: Process restarted (1.0)
        - Stage 4: Process verified online (1.0)
   
   Key Classes:
   - Task: Abstract base class
   - EasyTask: Log analysis implementation
   - MediumTask: Permission repair implementation
   - HardTask: Process recovery implementation

4. MAIN ENVIRONMENT (environment.py - 233 lines)
   ────────────────
   - OpenEnv-compliant main interface
   - Provides: reset(), step() interface
   - Manages:
     * Episode state and step counting
     * Task grading and scoring
     * Reward calculation
     * Terminal observation generation
   
   Key Methods:
   - reset(): Initialize environment, return initial observation
   - step(action): Execute command, return result with reward
   - get_state(): Get full internal state for debugging
   
   Reward Structure:
   - Step penalty: -0.01 (encourages efficiency)
   - Progress bonus: +0.5 per stage
   - Completion bonus: +0.5 (max 1.0 total per episode)
   
   Key Classes:
   - LinuxSREEnvironment: Main environment class

5. LLM AGENT INTEGRATION (agent.py - 198 lines)
   ──────────────────────
   - Interface for connecting language models to environment
   - Provides formatted observations for LLM consumption
   - Manages interaction history
   - Generates episode summaries
   - Provides system prompts for LLMs
   
   Key Classes:
   - AgentInterface: Main agent interaction class
   - AgentPromptGenerator: LLM prompt templates
   
   Usage Pattern:
   1. Create agent: agent = AgentInterface(task_difficulty="hard")
   2. Reset: initial_state = agent.reset()
   3. Execute: result = agent.execute_command("ps | grep postgres")
   4. Iterate until done
   5. Summarize: summary = agent.get_episode_summary()

6. REST API SERVER (server.py - 186 lines)
   ──────────────
   - FastAPI-based REST API
   - OpenEnv-compliant endpoints
   - Stateless architecture (scale horizontally)
   
   Key Endpoints:
   - POST   /api/v1/env/reset          → Create environment, get initial obs
   - POST   /api/v1/env/{id}/step      → Execute command
   - GET    /api/v1/tasks               → List available tasks
   - GET    /api/v1/tasks/{difficulty}  → Task details
   - GET    /api/v1/env/{id}/state      → Full state (debugging)
   - DELETE /api/v1/env/{id}            → Clean up environment
   - GET    /health                      → Health check
   
   Key Features:
   - Type-safe with Pydantic
   - Auto-documentation (Swagger UI at /docs)
   - Multiple env instances support
   - Async/concurrent handling

WORKFLOW EXAMPLE
================================================================================

SCENARIO: Agent solving Medium difficulty (Permission Repair)

1. RESET PHASE:
   Agent calls: POST /api/v1/env/reset
   Returns:
   {
     "env_id": "env_0",
     "observation": {
       "task_name": "Permission Repair",
       "task_description": "Make cleanup.sh executable",
       "filesystem": "cleanup.sh setup.sh",
       "processes": "..."
     },
     "info": {
       "instructions": "Use chmod to fix permissions...",
       "max_steps": 50
     }
   }

2. ANALYSIS PHASE:
   Agent thinks: "Task is to make cleanup.sh executable"
   Agent plans: "Need to check current permissions, then use chmod 0755"

3. EXECUTION PHASE - Step 1:
   Agent: POST /api/v1/env/env_0/step
   Command: "ls -la /home/user/scripts/cleanup.sh"
   Returns:
   {
     "observation": {...},
     "reward": -0.01,  (step penalty)
     "done": false,
     "info": {
       "task_score": 0.0,
       "task_metadata": {
         "stages_completed": [1]  (script located)
       }
     }
   }

4. EXECUTION PHASE - Step 2:
   Agent: POST /api/v1/env/env_0/step
   Command: "chmod 0755 /home/user/scripts/cleanup.sh"
   Returns:
   {
     "observation": {...},
     "reward": 0.99,  (progress bonus -0.01 + stage2 bonus 0.5 + stage3 bonus 0.5)
     "done": true,
     "info": {
       "task_score": 1.0,  (task completed!)
       "task_metadata": {
         "stages_completed": [1, 2, 3]
       }
     }
   }

5. SUMMARY:
   - Task completed in 2 steps
   - Efficiency score: 2/50 = 0.04 (very efficient)
   - Total reward: -0.01 + 0.99 = 0.98
   - Episode done

KEY ARCHITECTURAL DECISIONS
================================================================================

1. PURE PYTHON IMPLEMENTATION
   Why? 
   - No external dependencies on system state
   - Reproducible across all platforms
   - Safe (can't break anything)
   - Fast (no I/O delays)
   - Easy to inspect/debug

2. VIRTUAL FILESYSTEM OVER REAL FILES
   Why?
   - Perfect reproducibility (same input → same output)
   - No side effects on user's system
   - Instant reset to known state
   - Can easily verify correctness
   - No permission issues

3. RESTRICTED COMMAND SET
   Why?
   - Safe for agents to explore
   - Balances power vs predictability
   - Prevents catastrophic errors
   - Makes grading objective
   - Runs anywhere (no /dev/urandom dependencies)

4. STAGED GRADING (0.2 → 0.5 → 1.0)
   Why?
   - Shows learning progression
   - Better training signal for LLMs
   - Partial credit for partial solutions
   - Mirrors real SRE work (incremental fixes)
   - Judges like seeing progressive improvement

5. OpenEnv REST API
   Why?
   - Standard interface for benchmarking
   - Language/platform agnostic
   - Scales horizontally
   - Easy integration with agent frameworks
   - Judges expect standard APIs

GETTING STARTED
================================================================================

1. QUICK START (2 minutes):
   ```
   pip install -r requirements.txt
   python demo.py
   ```
   
   This shows all 3 tasks with example solutions.

2. RUN API SERVER (development):
   ```
   python -m uvicorn src.server:app --reload
   Open http://localhost:8000/docs for interactive API
   ```

3. RUN TESTS:
   ```
   pip install pytest
   pytest tests/ -v
   ```

4. DOCKER DEPLOYMENT:
   ```
   docker build -t linux-sre-env .
   docker run -p 8000:8000 linux-sre-env
   ```

5. USE IN YOUR OWN CODE:
   ```python
   from src.environment import LinuxSREEnvironment
   env = LinuxSREEnvironment(task_difficulty="hard")
   env.reset()
   result = env.step("ps | grep postgres")
   ```
